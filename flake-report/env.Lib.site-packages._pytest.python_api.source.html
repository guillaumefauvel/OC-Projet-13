516 def approx(expected, rel=None, abs=None, nan_ok: bool = False) -> ApproxBase:
517 """Assert that two numbers (or two ordered sequences of numbers) are equal to each other
518 within some tolerance.
519  
520 Due to the :std:doc:`tutorial/floatingpoint`, numbers that we
521 would intuitively expect to be equal are not always so::
522  
523 >>> 0.1 + 0.2 == 0.3
524 False
525  
526 This problem is commonly encountered when writing tests, e.g. when making
527 sure that floating-point values are what you expect them to be. One way to
528 deal with this problem is to assert that two floating-point numbers are
529 equal to within some appropriate tolerance::
530  
531 >>> abs((0.1 + 0.2) - 0.3) < 1e-6
532 True
533  
534 However, comparisons like this are tedious to write and difficult to
535 understand. Furthermore, absolute comparisons like the one above are
536 usually discouraged because there's no tolerance that works well for all
537 situations. ``1e-6`` is good for numbers around ``1``, but too small for
538 very big numbers and too big for very small ones. It's better to express
539 the tolerance as a fraction of the expected value, but relative comparisons
540 like that are even more difficult to write correctly and concisely.
541  
542 The ``approx`` class performs floating-point comparisons using a syntax
543 that's as intuitive as possible::
544  
545 >>> from pytest import approx
546 >>> 0.1 + 0.2 == approx(0.3)
547 True
548  
549 The same syntax also works for ordered sequences of numbers::
550  
551 >>> (0.1 + 0.2, 0.2 + 0.4) == approx((0.3, 0.6))
552 True
553  
554 ``numpy`` arrays::
555  
  • E501 Line too long (104 > 99 characters)
556 >>> import numpy as np # doctest: +SKIP
  • E501 Line too long (104 > 99 characters)
557 >>> np.array([0.1, 0.2]) + np.array([0.2, 0.4]) == approx(np.array([0.3, 0.6])) # doctest: +SKIP
558 True
559  
560 And for a ``numpy`` array against a scalar::
561  
562 >>> import numpy as np # doctest: +SKIP
563 >>> np.array([0.1, 0.2]) + np.array([0.2, 0.1]) == approx(0.3) # doctest: +SKIP
564 True
565  
566 Only ordered sequences are supported, because ``approx`` needs
567 to infer the relative position of the sequences without ambiguity. This means
568 ``sets`` and other unordered sequences are not supported.
569  
570 Finally, dictionary *values* can also be compared::
571  
572 >>> {'a': 0.1 + 0.2, 'b': 0.2 + 0.4} == approx({'a': 0.3, 'b': 0.6})
573 True
574  
575 The comparison will be true if both mappings have the same keys and their
576 respective values match the expected tolerances.
577  
578 **Tolerances**
579  
580 By default, ``approx`` considers numbers within a relative tolerance of
581 ``1e-6`` (i.e. one part in a million) of its expected value to be equal.
582 This treatment would lead to surprising results if the expected value was
583 ``0.0``, because nothing but ``0.0`` itself is relatively close to ``0.0``.
584 To handle this case less surprisingly, ``approx`` also considers numbers
585 within an absolute tolerance of ``1e-12`` of its expected value to be
586 equal. Infinity and NaN are special cases. Infinity is only considered
587 equal to itself, regardless of the relative tolerance. NaN is not
588 considered equal to anything by default, but you can make it be equal to
589 itself by setting the ``nan_ok`` argument to True. (This is meant to
590 facilitate comparing arrays that use NaN to mean "no data".)
591  
592 Both the relative and absolute tolerances can be changed by passing
593 arguments to the ``approx`` constructor::
594  
595 >>> 1.0001 == approx(1)
596 False
597 >>> 1.0001 == approx(1, rel=1e-3)
598 True
599 >>> 1.0001 == approx(1, abs=1e-3)
600 True
601  
602 If you specify ``abs`` but not ``rel``, the comparison will not consider
603 the relative tolerance at all. In other words, two numbers that are within
604 the default relative tolerance of ``1e-6`` will still be considered unequal
605 if they exceed the specified absolute tolerance. If you specify both
606 ``abs`` and ``rel``, the numbers will be considered equal if either
607 tolerance is met::
608  
609 >>> 1 + 1e-8 == approx(1)
610 True
611 >>> 1 + 1e-8 == approx(1, abs=1e-12)
612 False
613 >>> 1 + 1e-8 == approx(1, rel=1e-6, abs=1e-12)
614 True
615  
616 You can also use ``approx`` to compare nonnumeric types, or dicts and
617 sequences containing nonnumeric types, in which case it falls back to
618 strict equality. This can be useful for comparing dicts and sequences that
619 can contain optional values::
620  
621 >>> {"required": 1.0000005, "optional": None} == approx({"required": 1, "optional": None})
622 True
623 >>> [None, 1.0000005] == approx([None,1])
624 True
625 >>> ["foo", 1.0000005] == approx([None,1])
626 False
627  
628 If you're thinking about using ``approx``, then you might want to know how
629 it compares to other good ways of comparing floating-point numbers. All of
630 these algorithms are based on relative and absolute tolerances and should
631 agree for the most part, but they do have meaningful differences:
632  
633 - ``math.isclose(a, b, rel_tol=1e-9, abs_tol=0.0)``: True if the relative
634 tolerance is met w.r.t. either ``a`` or ``b`` or if the absolute
635 tolerance is met. Because the relative tolerance is calculated w.r.t.
636 both ``a`` and ``b``, this test is symmetric (i.e. neither ``a`` nor
637 ``b`` is a "reference value"). You have to specify an absolute tolerance
638 if you want to compare to ``0.0`` because there is no tolerance by
639 default. More information: :py:func:`math.isclose`.
640  
641 - ``numpy.isclose(a, b, rtol=1e-5, atol=1e-8)``: True if the difference
642 between ``a`` and ``b`` is less that the sum of the relative tolerance
643 w.r.t. ``b`` and the absolute tolerance. Because the relative tolerance
644 is only calculated w.r.t. ``b``, this test is asymmetric and you can
645 think of ``b`` as the reference value. Support for comparing sequences
646 is provided by :py:func:`numpy.allclose`. More information:
647 :std:doc:`numpy:reference/generated/numpy.isclose`.
648  
649 - ``unittest.TestCase.assertAlmostEqual(a, b)``: True if ``a`` and ``b``
650 are within an absolute tolerance of ``1e-7``. No relative tolerance is
651 considered , so this function is not appropriate for very large or very
652 small numbers. Also, it's only available in subclasses of ``unittest.TestCase``
653 and it's ugly because it doesn't follow PEP8. More information:
654 :py:meth:`unittest.TestCase.assertAlmostEqual`.

805 ``pytest.raises(Exc, func, match="passed on").match("my pattern")``.)
806  
807 .. currentmodule:: _pytest._code
808  
809 Use ``pytest.raises`` as a context manager, which will capture the exception of the given
810 type::
811  
812 >>> import pytest
813 >>> with pytest.raises(ZeroDivisionError):
814 ... 1/0
815  
816 If the code block does not raise the expected exception (``ZeroDivisionError`` in the example
817 above), or no exception at all, the check will fail instead.
818  
819 You can also use the keyword argument ``match`` to assert that the
820 exception matches a text or regex::
821  
822 >>> with pytest.raises(ValueError, match='must be 0 or None'):
823 ... raise ValueError("value must be 0 or None")
824  
825 >>> with pytest.raises(ValueError, match=r'must be \d+$'):
826 ... raise ValueError("value must be 42")
827  
828 The context manager produces an :class:`ExceptionInfo` object which can be used to inspect the
829 details of the captured exception::
830  
831 >>> with pytest.raises(ValueError) as exc_info:
832 ... raise ValueError("value must be 42")
833 >>> assert exc_info.type is ValueError
834 >>> assert exc_info.value.args[0] == "value must be 42"
835  
836 .. note::
837  
838 When using ``pytest.raises`` as a context manager, it's worthwhile to
839 note that normal context manager rules apply and that the exception
840 raised *must* be the final line in the scope of the context manager.
841 Lines of code after that, within the scope of the context manager will
842 not be executed. For example::
843  
844 >>> value = 15
845 >>> with pytest.raises(ValueError) as exc_info:
846 ... if value > 10:
847 ... raise ValueError("value must be <= 10")
848 ... assert exc_info.type is ValueError # this will not execute
849  
850 Instead, the following approach must be taken (note the difference in
851 scope)::
852  
853 >>> with pytest.raises(ValueError) as exc_info:
854 ... if value > 10:
855 ... raise ValueError("value must be <= 10")
856 ...
857 >>> assert exc_info.type is ValueError
858

860  
861 When using :ref:`pytest.mark.parametrize ref`
862 it is possible to parametrize tests such that
863 some runs raise an exception and others do not.
864  
865 See :ref:`parametrizing_conditional_raising` for an example.
866

868

870  
871 >>> raises(ZeroDivisionError, lambda: 1/0)

873

875  
876 >>> def f(x): return 1/x

880 >>> raises(ZeroDivisionError, f, x=0)

882  
883 The form above is fully supported but discouraged for new code because the
884 context manager form is regarded as more readable and less error-prone.
885
