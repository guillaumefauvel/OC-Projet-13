37 # We have two different kinds of echo server fixtures we use for testing. The
38 # first is a real server written using the stdlib ssl module and blocking
39 # sockets. It runs in a thread and we talk to it over a real socketpair(), to
40 # validate interoperability in a semi-realistic setting.
41 #
42 # The second is a very weird virtual echo server that lives inside a custom
43 # Stream class. It lives entirely inside the Python object space; there are no
44 # operating system calls in it at all. No threads, no I/O, nothing. It's
45 # 'send_all' call takes encrypted data from a client and feeds it directly into
46 # the server-side TLS state engine to decrypt, then takes that data, feeds it
47 # back through to get the encrypted response, and returns it from 'receive_some'. This
48 # gives us full control and reproducibility. This server is written using
49 # PyOpenSSL, so that we can trigger renegotiations on demand. It also allows
50 # us to insert random (virtual) delays, to really exercise all the weird paths
51 # in SSLStream's state engine.
52 #
53 # Both present a certificate for "trio-test-1.example.org".

505 # Note: this test fails horribly if we force TLS 1.2 and trigger a
506 # renegotiation at the beginning (e.g. by switching to the pyopenssl
507 # server). Usually the client crashes in SSLObject.write with "UNEXPECTED
508 # RECORD"; sometimes we get something more exotic like a SyscallError. This is
509 # odd because openssl isn't doing any syscalls, but so it goes. After lots of
510 # websearching I'm pretty sure this is due to a bug in OpenSSL, where it just
511 # can't reliably handle full-duplex communication combined with
512 # renegotiation. Nice, eh?
513 #
514 # https://rt.openssl.org/Ticket/Display.html?id=3712
515 # https://rt.openssl.org/Ticket/Display.html?id=2481
516 # http://openssl.6102.n7.nabble.com/TLS-renegotiation-failure-on-receiving-application-data-during-handshake-td48127.html
517 # https://stackoverflow.com/questions/18728355/ssl-renegotiation-with-full-duplex-socket-communication
518 #
519 # In some variants of this test (maybe only against the java server?) I've
520 # also seen cases where our send_all blocks waiting to write, and then our receive_some
521 # also blocks waiting to write, and they never wake up again. It looks like
522 # some kind of deadlock. I suspect there may be an issue where we've filled up
523 # the send buffers, and the remote side is trying to handle the renegotiation
524 # from inside a write() call, so it has a problem: there's all this application
525 # data clogging up the pipe, but it can't process and return it to the
526 # application because it's in write(), and it doesn't want to buffer infinite
527 # amounts of data, and... actually I guess those are the only two choices.
528 #
529 # NSS even documents that you shouldn't try to do a renegotiation except when
530 # the connection is idle:
531 #
532 # https://developer.mozilla.org/en-US/docs/Mozilla/Projects/NSS/SSL_functions/sslfnc.html#1061582
533 #
534 # I begin to see why HTTP/2 forbids renegotiation and TLS 1.3 removes it...
