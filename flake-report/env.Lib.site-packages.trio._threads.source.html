62 async def to_thread_run_sync(sync_fn, *args, cancellable=False, limiter=None):
63 """Convert a blocking operation into an async operation using a thread.
64  
65 These two lines are equivalent::
66  
67 sync_fn(*args)
68 await trio.to_thread.run_sync(sync_fn, *args)
69  
70 except that if ``sync_fn`` takes a long time, then the first line will
71 block the Trio loop while it runs, while the second line allows other Trio
72 tasks to continue working while ``sync_fn`` runs. This is accomplished by
73 pushing the call to ``sync_fn(*args)`` off into a worker thread.
74  
75 From inside the worker thread, you can get back into Trio using the
76 functions in `trio.from_thread`.
77  
78 Args:
79 sync_fn: An arbitrary synchronous callable.
80 *args: Positional arguments to pass to sync_fn. If you need keyword
81 arguments, use :func:`functools.partial`.
82 cancellable (bool): Whether to allow cancellation of this operation. See
83 discussion below.
84 limiter (None, or CapacityLimiter-like object):
85 An object used to limit the number of simultaneous threads. Most
86 commonly this will be a `~trio.CapacityLimiter`, but it could be
87 anything providing compatible
88 :meth:`~trio.CapacityLimiter.acquire_on_behalf_of` and
89 :meth:`~trio.CapacityLimiter.release_on_behalf_of` methods. This
90 function will call ``acquire_on_behalf_of`` before starting the
91 thread, and ``release_on_behalf_of`` after the thread has finished.
92  
93 If None (the default), uses the default `~trio.CapacityLimiter`, as
94 returned by :func:`current_default_thread_limiter`.
95  
96 **Cancellation handling**: Cancellation is a tricky issue here, because
97 neither Python nor the operating systems it runs on provide any general
98 mechanism for cancelling an arbitrary synchronous function running in a
99 thread. This function will always check for cancellation on entry, before
100 starting the thread. But once the thread is running, there are two ways it
101 can handle being cancelled:
102  
103 * If ``cancellable=False``, the function ignores the cancellation and
104 keeps going, just like if we had called ``sync_fn`` synchronously. This
105 is the default behavior.
106  
107 * If ``cancellable=True``, then this function immediately raises
108 `~trio.Cancelled`. In this case **the thread keeps running in
109 background** – we just abandon it to do whatever it's going to do, and
110 silently discard any return value or errors that it raises. Only use
111 this if you know that the operation is safe and side-effect free. (For
112 example: :func:`trio.socket.getaddrinfo` uses a thread with
113 ``cancellable=True``, because it doesn't really affect anything if a
114 stray hostname lookup keeps running in the background.)
115  
116 The ``limiter`` is only released after the thread has *actually*
117 finished – which in the case of cancellation may be some time after this
118 function has returned. If :func:`trio.run` finishes before the thread
119 does, then the limiter release method will never be called at all.
120  
121 .. warning::
122  
123 You should not use this function to call long-running CPU-bound
124 functions! In addition to the usual GIL-related reasons why using
125 threads for CPU-bound work is not very effective in Python, there is an
126 additional problem: on CPython, `CPU-bound threads tend to "starve out"
127 IO-bound threads <https://bugs.python.org/issue7946>`__, so using
128 threads for CPU-bound work is likely to adversely affect the main
129 thread running Trio. If you need to do this, you're better off using a
130 worker process, or perhaps PyPy (which still has a GIL, but may do a
131 better job of fairly allocating CPU time between threads).
132  
133 Returns:
134 Whatever ``sync_fn(*args)`` returns.
135  
136 Raises:
137 Exception: Whatever ``sync_fn(*args)`` raises.
138
