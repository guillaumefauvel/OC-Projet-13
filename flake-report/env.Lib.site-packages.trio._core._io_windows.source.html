33 # There's a lot to be said about the overall design of a Windows event
34 # loop. See
35 #
36 # https://github.com/python-trio/trio/issues/52
37 #
38 # for discussion. This now just has some lower-level notes:
39 #
40 # How IOCP fits together:
41 #
42 # The general model is that you call some function like ReadFile or WriteFile
43 # to tell the kernel that you want it to perform some operation, and the
44 # kernel goes off and does that in the background, then at some point later it
45 # sends you a notification that the operation is complete. There are some more
46 # exotic APIs that don't quite fit this pattern, but most APIs do.
47 #
48 # Each background operation is tracked using an OVERLAPPED struct, that
49 # uniquely identifies that particular operation.
50 #
51 # An "IOCP" (or "I/O completion port") is an object that lets the kernel send
52 # us these notifications -- basically it's just a kernel->userspace queue.
53 #
54 # Each IOCP notification is represented by an OVERLAPPED_ENTRY struct, which
55 # contains 3 fields:
56 # - The "completion key". This is an opaque integer that we pick, and use
57 # however is convenient.
58 # - pointer to the OVERLAPPED struct for the completed operation.
59 # - dwNumberOfBytesTransferred (an integer).
60 #
61 # And in addition, for regular I/O, the OVERLAPPED structure gets filled in
62 # with:
63 # - result code (named "Internal")
64 # - number of bytes transferred (named "InternalHigh"); usually redundant
65 # with dwNumberOfBytesTransferred.
66 #
67 # There are also some other entries in OVERLAPPED which only matter on input:
68 # - Offset and OffsetHigh which are inputs to {Read,Write}File and
69 # otherwise always zero
70 # - hEvent which is for if you aren't using IOCP; we always set it to zero.
71 #
72 # That describes the usual pattern for operations and the usual meaning of
73 # these struct fields, but really these are just some arbitrary chunks of
74 # bytes that get passed back and forth, so some operations like to overload
75 # them to mean something else.
76 #
77 # You can also directly queue an OVERLAPPED_ENTRY object to an IOCP by calling
78 # PostQueuedCompletionStatus. When you use this you get to set all the
79 # OVERLAPPED_ENTRY fields to arbitrary values.
80 #
81 # You can request to cancel any operation if you know which handle it was
82 # issued on + the OVERLAPPED struct that identifies it (via CancelIoEx). This
83 # request might fail because the operation has already completed, or it might
84 # be queued to happen in the background, so you only find out whether it
85 # succeeded or failed later, when we get back the notification for the
86 # operation being complete.
87 #
88 # There are three types of operations that we support:
89 #
90 # == Regular I/O operations on handles (e.g. files or named pipes) ==
91 #
92 # Implemented by: register_with_iocp, wait_overlapped
93 #
94 # To use these, you have to register the handle with your IOCP first. Once
95 # it's registered, any operations on that handle will automatically send
96 # completion events to that IOCP, with a completion key that you specify *when
97 # the handle is registered* (so you can't use different completion keys for
98 # different operations).
99 #
100 # We give these two dedicated completion keys: CKeys.WAIT_OVERLAPPED for
101 # regular operations, and CKeys.LATE_CANCEL that's used to make
102 # wait_overlapped cancellable even if the user forgot to call
103 # register_with_iocp. The problem here is that after we request the cancel,
104 # wait_overlapped keeps blocking until it sees the completion notification...
105 # but if the user forgot to register_with_iocp, then the completion will never
106 # come, so the cancellation will never resolve. To avoid this, whenever we try
107 # to cancel an I/O operation and the cancellation fails, we use
108 # PostQueuedCompletionStatus to send a CKeys.LATE_CANCEL notification. If this
109 # arrives before the real completion, we assume the user forgot to call
110 # register_with_iocp on their handle, and raise an error accordingly.
111 #
112 # == Socket state notifications ==
113 #
114 # Implemented by: wait_readable, wait_writable
115 #
116 # The public APIs that windows provides for this are all really awkward and
117 # don't integrate with IOCP. So we drop down to a lower level, and talk
118 # directly to the socket device driver in the kernel, which is called "AFD".
119 # Unfortunately, this is a totally undocumented internal API. Fortunately
120 # libuv also does this, so we can be pretty confident that MS won't break it
121 # on us, and there is a *little* bit of information out there if you go
122 # digging.
123 #
124 # Basically: we open a magic file that refers to the AFD driver, register the
125 # magic file with our IOCP, and then we can issue regular overlapped I/O
126 # operations on that handle. Specifically, the operation we use is called
127 # IOCTL_AFD_POLL, which lets us pass in a buffer describing which events we're
128 # interested in on a given socket (readable, writable, etc.). Later, when the
129 # operation completes, the kernel rewrites the buffer we passed in to record
130 # which events happened, and uses IOCP as normal to notify us that this
131 # operation has completed.
132 #
133 # Unfortunately, the Windows kernel seems to have bugs if you try to issue
134 # multiple simultaneous IOCTL_AFD_POLL operations on the same socket (see
135 # notes-to-self/afd-lab.py). So if a user calls wait_readable and
136 # wait_writable at the same time, we have to combine those into a single
137 # IOCTL_AFD_POLL. This means we can't just use the wait_overlapped machinery.
138 # Instead we have some dedicated code to handle these operations, and a
139 # dedicated completion key CKeys.AFD_POLL.
140 #
141 # Sources of information:
142 # - https://github.com/python-trio/trio/issues/52
143 # - Wepoll: https://github.com/piscisaureus/wepoll/
144 # - libuv: https://github.com/libuv/libuv/
145 # - ReactOS: https://github.com/reactos/reactos/
146 # - Ancient leaked copies of the Windows NT and Winsock source code:
147 # https://github.com/pustladi/Windows-2000/blob/661d000d50637ed6fab2329d30e31775046588a9/private/net/sockets/winsock2/wsp/msafd/select.c#L59-L655
148 # https://github.com/metoo10987/WinNT4/blob/f5c14e6b42c8f45c20fe88d14c61f9d6e0386b8e/private/ntos/afd/poll.c#L68-L707
149 # - The WSAEventSelect docs (this exposes a finer-grained set of events than
150 # select(), so if you squint you can treat it as a source of information on
151 # the fine-grained AFD poll types)
152 #
153 #
154 # == Everything else ==
155 #
156 # There are also some weirder APIs for interacting with IOCP. For example, the
157 # "Job" API lets you specify an IOCP handle and "completion key", and then in
158 # the future whenever certain events happen it sends uses IOCP to send a
159 # notification. These notifications don't correspond to any particular
160 # operation; they're just spontaneous messages you get. The
161 # "dwNumberOfBytesTransferred" field gets repurposed to carry an identifier
162 # for the message type (e.g. JOB_OBJECT_MSG_EXIT_PROCESS), and the
163 # "lpOverlapped" field gets repurposed to carry some arbitrary data that
164 # depends on the message type (e.g. the pid of the process that exited).
165 #
166 # To handle these, we have monitor_completion_key, where we hand out an
167 # unassigned completion key, let users set it up however they want, and then
168 # get any events that arrive on that key.
169 #
170 # (Note: monitor_completion_key is not documented or fully baked; expect it to
171 # change in the future.)

290 # AFD_POLL has a finer-grained set of events than other APIs. We collapse them
291 # down into Unix-style "readable" and "writable".
292 #
293 # Note: AFD_POLL_LOCAL_CLOSE isn't a reliable substitute for notify_closing(),
294 # because even if the user closes the socket *handle*, the socket *object*
295 # could still remain open, e.g. if the socket was dup'ed (possibly into
296 # another process). Explicitly calling notify_closing() guarantees that
297 # everyone waiting on the *handle* wakes up, which is what you'd expect.
298 #
299 # However, we can't avoid getting LOCAL_CLOSE notifications -- the kernel

320 # Annoyingly, while the API makes it *seem* like you can happily issue as many
321 # independent AFD_POLL operations as you want without them interfering with
322 # each other, in fact if you issue two AFD_POLL operations for the same socket
323 # at the same time with notification going to the same IOCP port, then Windows
324 # gets super confused. For example, if we issue one operation from
325 # wait_readable, and another independent operation from wait_writable, then
326 # Windows may complete the wait_writable operation when the socket becomes
327 # readable.
328 #
329 # To avoid this, we have to coalesce all the operations on a single socket

350 # The Windows kernel has a weird issue when using AFD handles. If you have N
351 # instances of wait_readable/wait_writable registered with a single AFD handle,
352 # then cancelling any one of them takes something like O(N**2) time. So if we
353 # used just a single AFD handle, then cancellation would quickly become very
354 # expensive, e.g. a program with N active sockets would take something like
355 # O(N**3) time to unwind after control-C. The solution is to spread our sockets
356 # out over multiple AFD handles, so that N doesn't grow too large for any
357 # individual handle.

506 def process_events(self, received):
507 for i in range(received):
508 entry = self._events[i]
509 if entry.lpCompletionKey == CKeys.AFD_POLL:
510 lpo = entry.lpOverlapped
511 op = self._afd_ops.pop(lpo)
512 waiters = op.waiters
513 if waiters.current_op is not op:
514 # Stale op, nothing to do
515 pass
516 else:
517 waiters.current_op = None
518 # I don't think this can happen, so if it does let's crash
519 # and get a debug trace.
520 if lpo.Internal != 0: # pragma: no cover
521 code = ntdll.RtlNtStatusToDosError(lpo.Internal)
522 raise_winerror(code)
523 flags = op.poll_info.Handles[0].Events
524 if waiters.read_task and flags & READABLE_FLAGS:
525 _core.reschedule(waiters.read_task)
526 waiters.read_task = None
527 if waiters.write_task and flags & WRITABLE_FLAGS:
528 _core.reschedule(waiters.write_task)
529 waiters.write_task = None
530 self._refresh_afd(op.poll_info.Handles[0].Handle)
531 elif entry.lpCompletionKey == CKeys.WAIT_OVERLAPPED:
532 # Regular I/O event, dispatch on lpOverlapped
533 waiter = self._overlapped_waiters.pop(entry.lpOverlapped)
534 overlapped = entry.lpOverlapped
535 transferred = entry.dwNumberOfBytesTransferred
536 info = CompletionKeyEventInfo(
537 lpOverlapped=overlapped, dwNumberOfBytesTransferred=transferred
538 )
539 _core.reschedule(waiter, Value(info))
540 elif entry.lpCompletionKey == CKeys.LATE_CANCEL:
541 # Post made by a regular I/O event's abort_fn
542 # after it failed to cancel the I/O. If we still
543 # have a waiter with this lpOverlapped, we didn't
544 # get the regular I/O completion and almost
545 # certainly the user forgot to call
546 # register_with_iocp.
547 self._posted_too_late_to_cancel.remove(entry.lpOverlapped)
548 try:
549 waiter = self._overlapped_waiters.pop(entry.lpOverlapped)
550 except KeyError:
551 # Looks like the actual completion got here before this
552 # fallback post did -- we're in the "expected" case of
553 # too-late-to-cancel, where the user did nothing wrong.
554 # Nothing more to do.
555 pass
556 else:
557 exc = _core.TrioInternalError(
558 "Failed to cancel overlapped I/O in {} and didn't "
559 "receive the completion either. Did you forget to "
560 "call register_with_iocp()?".format(waiter.name)
561 )
562 # Raising this out of handle_io ensures that
563 # the user will see our message even if some
564 # other task is in an uncancellable wait due
565 # to the same underlying forgot-to-register
566 # issue (if their CancelIoEx succeeds, we
567 # have no way of noticing that their completion
568 # won't arrive). Unfortunately it loses the
569 # task traceback. If you're debugging this
570 # error and can't tell where it's coming from,
571 # try changing this line to
572 # _core.reschedule(waiter, outcome.Error(exc))
573 raise exc
574 elif entry.lpCompletionKey == CKeys.FORCE_WAKEUP:
575 pass
576 else:
577 # dispatch on lpCompletionKey
578 queue = self._completion_key_queues[entry.lpCompletionKey]
579 overlapped = int(ffi.cast("uintptr_t", entry.lpOverlapped))
580 transferred = entry.dwNumberOfBytesTransferred
581 info = CompletionKeyEventInfo(
582 lpOverlapped=overlapped, dwNumberOfBytesTransferred=transferred
583 )
584 queue.put_nowait(info)

603 def _refresh_afd(self, base_handle):
604 waiters = self._afd_waiters[base_handle]
605 if waiters.current_op is not None:
606 afd_group = waiters.current_op.afd_group
607 try:
608 _check(
609 kernel32.CancelIoEx(
610 afd_group.handle, waiters.current_op.lpOverlapped
611 )
612 )
613 except OSError as exc:
614 if exc.winerror != ErrorCodes.ERROR_NOT_FOUND:
615 # I don't think this is possible, so if it happens let's
616 # crash noisily.
617 raise # pragma: no cover
618 waiters.current_op = None
619 afd_group.size -= 1
620 self._vacant_afd_groups.add(afd_group)
621  
622 flags = 0
623 if waiters.read_task is not None:
624 flags |= READABLE_FLAGS
625 if waiters.write_task is not None:
626 flags |= WRITABLE_FLAGS
627  
628 if not flags:
629 del self._afd_waiters[base_handle]
630 else:
631 try:
632 afd_group = self._vacant_afd_groups.pop()
633 except KeyError:
634 afd_group = AFDGroup(0, _afd_helper_handle())
635 self._register_with_iocp(afd_group.handle, CKeys.AFD_POLL)
636 self._all_afd_handles.append(afd_group.handle)
637 self._vacant_afd_groups.add(afd_group)
638  
639 lpOverlapped = ffi.new("LPOVERLAPPED")
640  
641 poll_info = ffi.new("AFD_POLL_INFO *")
642 poll_info.Timeout = 2**63 - 1 # INT64_MAX
643 poll_info.NumberOfHandles = 1
644 poll_info.Exclusive = 0
645 poll_info.Handles[0].Handle = base_handle
646 poll_info.Handles[0].Status = 0
647 poll_info.Handles[0].Events = flags
648  
649 try:
650 _check(
651 kernel32.DeviceIoControl(
652 afd_group.handle,
653 IoControlCodes.IOCTL_AFD_POLL,
654 poll_info,
655 ffi.sizeof("AFD_POLL_INFO"),
656 poll_info,
657 ffi.sizeof("AFD_POLL_INFO"),
658 ffi.NULL,
659 lpOverlapped,
660 )
661 )
662 except OSError as exc:
663 if exc.winerror != ErrorCodes.ERROR_IO_PENDING:
664 # This could happen if the socket handle got closed behind
665 # our back while a wait_* call was pending, and we tried
666 # to re-issue the call. Clear our state and wake up any
667 # pending calls.
668 del self._afd_waiters[base_handle]
669 # Do this last, because it could raise.
670 wake_all(waiters, exc)
671 return
672 op = AFDPollOp(lpOverlapped, poll_info, waiters, afd_group)
673 waiters.current_op = op
674 self._afd_ops[lpOverlapped] = op
675 afd_group.size += 1
676 if afd_group.size >= MAX_AFD_GROUP_SIZE:
677 self._vacant_afd_groups.remove(afd_group)

679 async def _afd_poll(self, sock, mode):
680 base_handle = _get_base_socket(sock)
681 waiters = self._afd_waiters.get(base_handle)
682 if waiters is None:
683 waiters = AFDWaiters()
684 self._afd_waiters[base_handle] = waiters
685 if getattr(waiters, mode) is not None:
686 raise _core.BusyResourceError
687 setattr(waiters, mode, _core.current_task())
688 # Could potentially raise if the handle is somehow invalid; that's OK,
689 # we let it escape.
690 self._refresh_afd(base_handle)
691  
692 def abort_fn(_):
693 setattr(waiters, mode, None)
694 self._refresh_afd(base_handle)
695 return _core.Abort.SUCCEEDED
696  
697 await _core.wait_task_rescheduled(abort_fn)

723 @_public
724 async def wait_overlapped(self, handle, lpOverlapped):
725 handle = _handle(handle)
726 if isinstance(lpOverlapped, int):
727 lpOverlapped = ffi.cast("LPOVERLAPPED", lpOverlapped)
728 if lpOverlapped in self._overlapped_waiters:
729 raise _core.BusyResourceError(
730 "another task is already waiting on that lpOverlapped"
731 )
732 task = _core.current_task()
733 self._overlapped_waiters[lpOverlapped] = task
734 raise_cancel = None
735  
736 def abort(raise_cancel_):
737 nonlocal raise_cancel
738 raise_cancel = raise_cancel_
739 try:
740 _check(kernel32.CancelIoEx(handle, lpOverlapped))
741 except OSError as exc:
742 if exc.winerror == ErrorCodes.ERROR_NOT_FOUND:
743 # Too late to cancel. If this happens because the
744 # operation is already completed, we don't need to do
745 # anything; we'll get a notification of that completion
746 # soon. But another possibility is that the operation was
747 # performed on a handle that wasn't registered with our
748 # IOCP (ie, the user forgot to call register_with_iocp),
749 # in which case we're just never going to see the
750 # completion. To avoid an uncancellable infinite sleep in
751 # the latter case, we'll PostQueuedCompletionStatus here,
752 # and if our post arrives before the original completion
753 # does, we'll assume the handle wasn't registered.
754 _check(
755 kernel32.PostQueuedCompletionStatus(
756 self._iocp, 0, CKeys.LATE_CANCEL, lpOverlapped
757 )
758 )
759 # Keep the lpOverlapped referenced so its address
760 # doesn't get reused until our posted completion
761 # status has been processed. Otherwise, we can
762 # get confused about which completion goes with
763 # which I/O.
764 self._posted_too_late_to_cancel.add(lpOverlapped)
765 else: # pragma: no cover
766 raise _core.TrioInternalError(
767 "CancelIoEx failed with unexpected error"
768 ) from exc
769 return _core.Abort.FAILED
770  
771 info = await _core.wait_task_rescheduled(abort)
772 if lpOverlapped.Internal != 0:
773 # the lpOverlapped reports the error as an NT status code,
774 # which we must convert back to a Win32 error code before
775 # it will produce the right sorts of exceptions
776 code = ntdll.RtlNtStatusToDosError(lpOverlapped.Internal)
777 if code == ErrorCodes.ERROR_OPERATION_ABORTED:
778 if raise_cancel is not None:
779 raise_cancel()
780 else:
781 # We didn't request this cancellation, so assume
782 # it happened due to the underlying handle being
783 # closed before the operation could complete.
784 raise _core.ClosedResourceError("another task closed this resource")
785 else:
786 raise_winerror(code)
787 return info

789 async def _perform_overlapped(self, handle, submit_fn):
790 # submit_fn(lpOverlapped) submits some I/O
791 # it may raise an OSError with ERROR_IO_PENDING
792 # the handle must already be registered using
793 # register_with_iocp(handle)
794 # This always does a schedule point, but it's possible that the
795 # operation will not be cancellable, depending on how Windows is
796 # feeling today. So we need to check for cancellation manually.
797 await _core.checkpoint_if_cancelled()
798 lpOverlapped = ffi.new("LPOVERLAPPED")
799 try:
800 submit_fn(lpOverlapped)
801 except OSError as exc:
802 if exc.winerror != ErrorCodes.ERROR_IO_PENDING:
803 raise
804 await self.wait_overlapped(handle, lpOverlapped)
805 return lpOverlapped
